# UI プロトタイプ

1 つのルート上に**大胆に異なる複数の UI バリエーション**を生成し、フローティングボトムバーで切り替えられるようにする。ユーザーはブラウザでバリアントを行き来し、1 つを選ぶ（または各案の良いところを盗む）。残りは捨てる。

疑問が見た目ではなくロジック / 状態についてなら — 分岐違い。[LOGIC.md](LOGIC.md) を使うこと。

## この形が正しいとき

- 「このページはどう見えるべきか？」
- 「このダッシュボード、決める前にいくつか案を見たい。」
- 「設定画面のレイアウトを別案で試して。」
- 頭の中の曖昧なモックアップ 3 案の間で丸一日悩みそうなとき全般。

## 2 つのサブ形 — サブ形 A を強く推奨

UI プロトタイプは、**アプリの残りの部分と隣接している**ときの方がはるかに判断しやすい — 本物のヘッダー、本物のサイドバー、本物のデータ、本物の情報密度。単独の使い捨てルートは真空だ：どのバリアントも単体ではそれなりに見えてしまう。バリアントを載せられる既存ページがありそうなら、常にサブ形 A をデフォルトにする。サブ形 B に手を伸ばすのは、プロトタイプに本当に近い置き場所がないときだけ。

### サブ形 A — 既存ページへの調整（推奨）

ルートはすでに存在する。バリアントは**同じルート上**に描画し、URL の検索パラメータ `?variant=` で出し分ける。既存のデータ取得・パラメータ・認証はすべてそのまま — 描画部分だけを差し替える。これがデフォルト。特別な理由がない限りこれを選ぶ。

まだページを持たないものでも、*自然に既存ページの中に住めるもの*（ダッシュボードの新しいセクション、設定画面の新しいカード、既存フローの新しいステップ）なら — それもサブ形 A。ホストページの中にバリアントをマウントする。

### サブ形 B — 新しいページ（最終手段）

プロトタイプ対象が本当にどの既存ページの中にも住めないとき（まったく新しいトップレベルの画面、どこにも埋め込めないフローなど）にだけ使う。

**使い捨てルート**を、プロジェクト既存のルーティング規約に従って作る — 新しいトップレベル構造を発明しない。明らかにプロトタイプと分かる名前を付ける（パスやファイル名に `prototype` を含めるなど）。`?variant=` のパターンは同じ。

サブ形 B に決める前に確認：本当にこれを埋め込める既存ページはないか？ 空っぽのルートは、埋まったルートなら露呈させる設計上の問題を隠してしまう。

どちらのサブ形でも、フローティングボトムバーは同一。

## 手順

### 1. 疑問を明文化し、N を決める

デフォルトは**3 バリアント**。5 を超えると「大胆に異なる」ではなくノイズになる — 上限は 5。

計画を 1 行で、プロトタイプの置き場所かファイル先頭のコメントに書く：

> 「設定ページの 3 バリアント。既存の `/settings` ルート上で `?variant=` により切り替え。」

ユーザーがその場で反論できてもできなくても、これは機能する。

### 2. 大胆に異なるバリアントを生成する

各バリアントを下書きする。それぞれ次を守ること：

- ページの目的と、そのページがアクセスできるデータ。
- プロジェクトのコンポーネントライブラリ / スタイリングシステム（TailwindCSS、shadcn、MUI、素の CSS、何であれ）。
- 明確にエクスポートされたコンポーネント名。例：`VariantA`、`VariantB`、`VariantC`。

バリアントは**構造的に異なる**こと — 異なるレイアウト、異なる情報の階層、異なる主要な操作導線。色違いではない。微調整しただけのカードグリッド 3 枚は UI プロトタイプではなく壁紙だ。2 案が似すぎたら、片方を「カードグリッド禁止」の明示的な縛りで作り直す。

### 3. 配線する

ルート上に切り替えコンポーネントを 1 つ作る：

```tsx
// 疑似コード — プロジェクトのフレームワークに合わせること
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

サブ形 A（既存ページ）：既存のデータ取得はすべてスイッチャーの上に残す。バリアントごとに変わるのは描画サブツリーだけ。

サブ形 B（新ページ）：`/prototype/<name>` 配下の使い捨てルートに同じスイッチャーをマウントする。

### 4. フローティングスイッチャーを作る

画面下部中央に固定表示される小さなバー。3 つの要素：

- **左矢印** — 前のバリアントへ（末尾から先頭へ循環）。
- **バリアントラベル** — 現在のバリアントのキーと、バリアントが名前をエクスポートしていればそれも表示。例：`B — サイドバー型レイアウト`。
- **右矢印** — 次のバリアントへ（循環）。

振る舞い：

- 矢印クリックで URL の検索パラメータを更新する（フレームワークのルーターを使う — Next なら `router.replace`、React Router なら `navigate` 等）。これでバリアントは共有可能かつリロード耐性を持つ。
- キーボード：`←` と `→` でも循環。ただし `<input>`、`<textarea>`、`[contenteditable]` にフォーカスがあるときは矢印キーを横取りしない。
- 評価対象のデザインの一部と誤認されないよう、ページから視覚的に区別する（高コントラストのピル型、控えめな影など）。
- 本番ビルドでは非表示 — `process.env.NODE_ENV !== 'production'` 相当のチェックでガードし、プロトタイプの誤マージでバーがユーザーに出荷されないようにする。

スイッチャーは共有コンポーネント 1 つにまとめ、両サブ形で再利用できるようにする。プロジェクトで共有 UI が置かれる場所に置くこと。

### 5. 引き渡す

URL（と `?variant=` のキー一覧）を提示する。ユーザーは手が空いたときに見て回る。面白いフィードバックはたいてい**「B のヘッダーと C のサイドバーを組み合わせたい」** — それこそが本当に欲しいデザインだ。

### 6. 答えを記録して片付ける

勝者が決まったら、どれをなぜ選んだかを書き残す（コミットメッセージ、ADR、Issue、あるいは無人実行中でユーザーの返答待ちならプロトタイプの隣の `NOTES.md`）。それから：

- **サブ形 A** — 負けたバリアントとスイッチャーを削除し、勝者を既存ページに畳み込む。
- **サブ形 B** — 勝ったバリアントを本物のルートに昇格させ、使い捨てルートとスイッチャーを削除する。

バリアントのコンポーネントやスイッチャーを放置しないこと。すぐ腐って次の読者を混乱させる。

## アンチパターン

- **色や文言しか違わないバリアント。** それは微調整であってプロトタイプではない。本物のバリアントは構造について意見が割れる。
- **バリアント間でコードを共有しすぎる。** `<Header>` の共有はいい。`<Layout>` の共有は本末転倒。各バリアントはレイアウトごと捨てられる自由を持つべきだ。
- **バリアントを本物の更新処理に繋ぐ。** 読み取り専用のプロトタイプでいい。更新が必要ならスタブに向ける — 疑問は「どう見えるべきか」であって「バックエンドが動くか」ではない。
- **プロトタイプをそのまま本番に昇格させる。** バリアントのコードはプロトタイプの制約（テストなし、最小限のエラーハンドリング）の下で書かれている。畳み込むときにきちんと書き直すこと。
