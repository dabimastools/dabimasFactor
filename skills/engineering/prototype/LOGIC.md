# ロジック・プロトタイプ

ユーザーが手でステートモデルを操作できる、小さな対話型ターミナルアプリ。疑問が**ビジネスロジック、状態遷移、データの形**に関するものであるときに使う — 紙の上では妥当に見えるのに、実際のケースを通してみて初めて違和感に気づく類のもの。

## この形が正しいとき

- 「このステートマシン、X の後に Y が来る端ケースを扱えるか自信がない。」
- 「このデータモデルで、〜のケースを本当に表現できるのか…」
- 「実装する前に API がどんな形になるべきか感触を掴みたい。」
- ユーザーが**ボタンを押して状態が変わるのを見たい**もの全般。

疑問が「どう見えるべきか」なら — 分岐違い。[UI.md](UI.md) を使うこと。

## 手順

### 1. 疑問を明文化する

コードを書く前に、どんなステートモデルを、どんな疑問のためにプロトタイプするのかを書き出す。1 段落で、プロトタイプの README かファイル先頭のコメントに。間違った疑問に答えるロジック・プロトタイプは純粋な無駄だ — 後から検証できるよう疑問を明示的にしておくこと。ユーザーが今見ていようと、離席後に戻ってこようと。

### 2. 言語を選ぶ

ホストプロジェクトが使っているものを使う。プロジェクトに明白なランタイムがない場合（ドキュメントだけのリポジトリなど）は尋ねる。

ツーリングはプロジェクトの既存の慣習に合わせる — プロトタイプのためだけに新しいパッケージマネージャやランタイムを追加しない。

### 3. ロジックをポータブルなモジュールに分離する

実際のロジック — 疑問に答えている部分 — は、後で持ち上げて本物のコードベースに落とし込めるような、小さくて純粋なインターフェースの背後に置く。周りの TUI は使い捨てだが、ロジックのモジュールはそうであってはならない。

正しい形は疑問によって決まる：

- **純粋なリデューサー** — `(state, action) => state`。アクションが離散的なイベントで、状態が単一の値のときに良い。
- **ステートマシン** — 明示的な状態と遷移。「今どのアクションがそもそも合法なのか」自体が疑問の一部のときに良い。
- **プレーンなデータ型に対する少数の純粋関数**。暗黙の「現在の状態」がなく、変換だけがあるときに良い。
- **明確なメソッド表面を持つクラスまたはモジュール** — ロジックが本当に継続的な内部状態を所有しているとき。

TUI に配線しやすい形ではなく、問われている疑問に最も合う形を選ぶこと。純粋に保つ：I/O なし、ターミナルコードなし、制御フローのための `console.log` なし。TUI がロジックを import して呼ぶ。逆方向には何も流さない。

これが、プロトタイプの寿命を超えて役立つ理由だ。疑問に答えが出たら、検証済みのリデューサー / マシン / 関数群は本物のモジュールに持ち上げられる — TUI の殻は削除される。

### 4. 状態を露出する最小の TUI を作る

**軽量 TUI** として作る — 毎ティック、画面をクリアして（`console.clear()` / `print("\033[2J\033[H")` / 相当）、フレーム全体を再描画する。ユーザーには常に 1 つの安定したビューが見えるべきで、延々と伸びるスクロールバックではない。

各フレームは 2 部構成、この順で：

1. **現在の状態**。読みやすく、diff しやすく整形する（1 行 1 フィールド、または整形済み JSON）。フィールド名やセクション見出しには**太字**、重要度の低い文脈（タイムスタンプ、ID、導出値）には**薄字（dim）**を使う。素の ANSI エスケープコードで構わない — `\x1b[1m` 太字、`\x1b[2m` 薄字、`\x1b[0m` リセット。プロジェクトにすでに入っていない限り、スタイリング用ライブラリを入れる必要はない。
2. **キーボードショートカット**を下部に列挙：`[a] ユーザー追加  [d] ユーザー削除  [t] 時計を進める  [q] 終了`。キーを太字、説明を薄字に（またはその逆に）— 読みやすい方で。

振る舞い：

1. **状態を初期化** — メモリ上の単一のオブジェクト / 構造体。起動時に最初のフレームを描画。
2. **キー入力（または 1 行）を 1 つずつ読み**、状態を変更するハンドラにディスパッチ。
3. アクションのたびにフレーム全体を**再描画** — 追記ではなく置き換え。
4. **終了までループ。**

フレーム全体が 1 画面に収まるべき。

### 5. 1 コマンドで実行可能にする

プロジェクト既存のタスクランナー（`package.json` の scripts、`Makefile`、`justfile`、`pyproject.toml`）にスクリプトを追加する。ユーザーが `pnpm run <プロトタイプ名>` 相当で実行できること — パスを覚える必要があってはならない。

ホストプロジェクトにタスクランナーがなければ、プロトタイプの README の先頭にコマンドを書いておく。

### 6. 引き渡す

ユーザーに実行コマンドを伝える。操作は本人がする。面白いのは「待って、それはあり得ないはずだ」「あれ、X は違うと思ってた」という瞬間 — それが_アイデアの_バグであり、それこそが目的だ。新しいアクションが欲しいと言われたら追加する。プロトタイプは進化する。

### 7. 答えを記録する

プロトタイプが役目を終えたら、残す価値があるのは疑問への答えだけだ。ユーザーがいるなら、何を学んだか尋ねる。いないなら、プロトタイプの隣に `NOTES.md` を残し、プロトタイプが削除される前に答えを書き込めるようにする（セッションを見ていたなら自分で書き込んでもよい）。

## アンチパターン

- **テストを足さない。** テストが必要なプロトタイプは、もはやプロトタイプではない。
- **本物のデータベースに繋がない。** 疑問が永続化そのものについてでない限り、インメモリのストアを使う。
- **一般化しない。** 「後で X をサポートしたくなったら」は禁止。プロトタイプは 1 つの疑問に答える。
- **ロジックと TUI を混ぜない。** リデューサー / ステートマシンが `console.log`、プロンプト、ターミナルのエスケープコードを参照していたら、もうポータブルではない。TUI は純粋なモジュールを包む薄い殻に保つこと。
- **TUI の殻を本番に持ち込まない。** 殻はターミナルから手で操作されるために最適化されている。残す価値があるのは、その背後のロジックモジュールだ。
